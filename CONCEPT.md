## 🎯 プロダクトビジョン

### ミッション

**「飲み会の幹事業務を、もっと楽に、もっと楽しく」**

飲み会の幹事は、本来楽しいはずのイベントを企画する役割なのに、集金や日程調整といった面倒な作業に追われてしまいます。KANJYは、そんな幹事さんの負担を減らし、本来の「みんなで楽しむ」ことに集中できるようサポートします。

### ビジョン

**「飲み会といえばKANJY」**

飲み会の幹事を任された時、誰もが真っ先に「KANJYを使おう」と思うアプリになる。

---

## 🍻 なぜ「飲み会」特化なのか

### 2025年11月22日の重要な決定

開発途中で「イベント管理アプリ」への拡張も検討しましたが、最終的に**飲み会特化**で進めることを決定しました。

#### 決定の背景

1. **ターゲットの明確化**
   - 「イベント管理」は範囲が広すぎる
   - 飲み会に絞ることで、誰のための何のアプリか明確になる

2. **競合との差別化**
   - イベント管理アプリは既に多数存在
   - 「飲み会専門」というポジションは空いている

3. **機能開発の方向性**
   - 飲み会ならではの機能に集中できる
   - 二次会、役職による金額差など、飲み会特有のニーズに応えられる

4. **マーケティングのしやすさ**
   - 「飲み会の幹事になったらKANJY」というメッセージが明確
   - AppStoreで「飲み会 幹事」で検索されやすい

#### 重要な方針

**UI表記は「飲み会」、内部実装は「Event」**

- ユーザーに見える部分：「飲み会を作成」「飲み会の集金」
- コード内部：`ScheduleEvent`, `EventDTO` などは技術的な名称として使用
- この使い分けにより、柔軟性を保ちつつ明確なメッセージを発信

## 👥 ターゲットユーザー

### プライマリターゲット

**20代〜30代の若手・中堅社員**

- 会社の飲み会幹事を任されることが多い
- スマホネイティブ世代
- 効率化ツールに抵抗がない

### ペルソナ例

**田中太郎（28歳・営業職）**
```
状況:
- 部署の忘年会の幹事を任された（10人規模）
- 上司は多めに払うべきだが、計算が面倒
- 誰が払ったか分からなくなることがある

悩み:
- Excelで計算するのが面倒
- LINEで集金状況を管理しているが、メッセージが流れる
- 日程調整と集金が別々で管理が大変

KANJYで解決:
- 役職ごとの金額を自動計算
- 集金状況が一目で分かる
- スケジュール調整から集金まで一元管理
```

---

## 💡 コア価値

### 1. 集金管理の簡単さ

**「誰が払ったか」を絶対に見失わない**

- プログレスバーで視覚的に表示
- ドットインジケーターで人数確認
- 未払い人数を常に表示

### 2. 役職による傾斜計算

**「上司は多め、新人は少なめ」を自動計算**

- 部長×2.0、課長×1.5、一般×1.0、新人×0.5
- カスタム役職も作成可能
- 固定金額も設定可能

### 3. 一元管理

**「あっちこっち探さなくていい」**

- 日程調整 → 参加者確定 → 集金 → 完了
- 全てのデータが1つのアプリに
- 過去の飲み会も簡単に参照

## 🔄 コア・ワークフロー（重要）

ユーザー（幹事）の体験フローは以下の通りです。この流れを理解することが開発において最も重要です。

1. **日程候補の作成**
   - 幹事がイベントを作成し、複数の開催候補日時を提案します。

2. **スケジュール調整（投票）**
   - 「調整さん」のようにURLを共有し、参加者に候補日への出欠（◯/△/✕）を投票してもらいます。

3. **開催日時の決定（ここが重要！）**
   - 投票結果（参加率の高い日や必須メンバーの回答）を見て、**幹事が最終的な開催日時を1つ選び決定します。**
   - 日時を決定＝参加者の絞り込み完了です。

4. **集金対象の確定**
   - **決定された開催日時に「参加（◯）」と回答しているメンバーだけが、飲み会後の集金リストに表示されます。**
   - 欠席者や、他の候補日なら行けた人には請求が発生しません。

## 🎨 UI/UXの原則

### 1. 集金状況を最優先で表示

ホーム画面で最も目立つべき情報は「集金状況」

```
良い例:
┌─────────────────────────┐
│ 🍻 忘年会              │
│ 集金 ●●●●●●●○○○    │ ← これ！
│ 7/10人 (未: 3人)      │
└─────────────────────────┘

悪い例:
┌─────────────────────────┐
│ 🍻 忘年会              │
│ 12/15 | 10人          │
│ ● ステータス          │ ← 何のステータス？
└─────────────────────────┘
```

### 2. シンプルさ > 多機能

統計情報やグラフは「あると嬉しい」程度。必須ではない。

削除した要素：
- ❌ 統計サマリーカード（総プラン数、総参加者数など）
- ❌ 長い説明文
- ❌ 複雑なグラフ

残した要素：
- ✅ 飲み会リスト
- ✅ 集金進捗
- ✅ 新規作成ボタン

### 3. 親しみやすさ

堅苦しくないカジュアルなトーン

```
良い例:
- 「新しい飲み会を作成」
- 「集金完了！」
- 「お疲れ様でした🎉」

悪い例:
- 「イベントの新規登録」
- 「決済処理が完了しました」
- 「ありがとうございました」
```

### 4. 直感的な操作

説明不要で使えるUI

- タップしたら何が起こるか明確
- アニメーションでフィードバック
- ハプティックで触覚フィードバック

## 🚀 機能開発の優先順位

### 最優先（Phase 1）

1. **集金管理機能**
   - ✅ 参加者管理
   - ✅ 役職設定
   - ✅ 金額計算
   - ✅ 集金状況の可視化

2. **スケジュール調整**
   - ✅ 候補日時の設定
   - ✅ Web経由での回答
   - ✅ 出欠管理

### 高優先（Phase 2）- 飲み会特化機能

1. **二次会管理** 🥇
   ```
   理由: 飲み会ならではの悩み
   実装: 中
   価値: 高
   
   機能:
   - 一次会・二次会を分けて管理
   - 両方参加する人は合計金額を自動計算
   - 二次会のみ参加者も設定可能
   ```

2. **お酒を飲む/飲まない設定** 🥈
   ```
   理由: よくあるリクエスト
   実装: 低
   価値: 高
   
   機能:
   - 飲む/飲まないフラグ
   - 飲まない人は0.8倍など
   - カスタマイズ可能
   ```

3. **居酒屋予約情報** 🥉
   ```
   理由: 情報を一元管理
   実装: 低
   価値: 中
   
   機能:
   - 店名・住所・電話番号
   - 予約時間・人数
   - 地図リンク
   ```

### 中優先（Phase 3）

- 飲み会テンプレート
- 割り勘計算の最適化
- 領収書スキャン（OCR）

### 低優先（Phase 4）

- 思い出機能
- ソーシャル機能
- コミュニティ形成

## ❌ やらないこと（重要）

### 1. 汎用イベント管理アプリ化

```
NG: 「イベント」という表記に統一
OK: 「飲み会」で統一（一部内部実装は Event でOK）

理由:
- ターゲットがぼやける
- 競合との差別化ができない
- 「何でもできる」は「何も得意じゃない」
```

### 2. 本格的な会計機能

```
NG:
- 経費精算
- 領収書管理
- 複式簿記

理由:
- 幹事の「ちょっとした集金」のためのアプリ
- 会計ソフトと競合する必要はない
```

### 3. SNS化

```
NG:
- ユーザー間のフォロー機能
- タイムライン
- 公開プロフィール

理由:
- プライバシーの問題
- 本来の価値（集金管理）から離れる
- SNS疲れの人も多い
```

このアプリを作る上で、常に心に留めておくべきこと：

1. **ユーザーは幹事さん**
   - 面倒な作業を任された人
   - できるだけ楽にしてあげたい

2. **飲み会は楽しいもの**

   - UIも楽しく、親しみやすく

3. **シンプルが一番**
   - 機能が多い = 良いではない
   - 必要最小限で最大の価値

4. **飲み会特化を貫く**
   - 汎用化の誘惑に負けない
   - ニッチで勝つ
